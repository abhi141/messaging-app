# Real-Time Messaging Application

A robust, full-stack 1-to-1 direct messaging platform built with a focus on real-time performance, request traceability, and modern UI/UX.

---

## 🚀 Getting Started

### Prerequisites
* **Node.js** (v18+)
* **MongoDB** (running locally on port 27017 or via URI)
* **npm** or **yarn**

### 1. Backend Setup
```bash
cd server
npm install
# Ensure you have a .env file (copy from .env.example)
npm run dev
```
*   **API Server:** [http://localhost:4000](http://localhost:4000)
*   **Swagger Docs:** [http://localhost:4000/docs](http://localhost:4000/docs)

### 2. Frontend Setup
```bash
cd client
npm install
npm run dev
```
*   **Web App:** [http://localhost:5173](http://localhost:5173)

## 🛠 Coding Standards & Architecture

This project follows strict engineering standards to ensure maintainability and scalability:

*   **Clean Architecture:** Separated into Controllers (routing), Services (business logic), and Repositories (data access).
*   **TypeScript-First:** Comprehensive typing across the full stack to catch errors at compile-time.
*   **Request Traceability:** Every request/socket event is tagged with a `correlationId`, which is propagated through service layers for granular debugging in logs.
*   **Normalized API Responses:** To ensure data consistency, the backend dynamically populates user profile data (like display names) in message responses rather than storing redundant data in the Message document.
*   **Deterministic Conversation Routing:** Conversations are identified via a deterministic `conversationId` generated by sorting and joining participant IDs (`user1_user2`), simplifying queries.

## ⚠️ Limitations

### 1. Functional Limitations
*   **Group Chats:** Currently only supports 1-to-1 direct messaging.
*   **Media Attachments:** The UI includes placeholders for images, but actual file storage (e.g., S3 integration) is not yet implemented.
*   **Offline Notifications:** Real-time messages are only received when both parties are connected via WebSockets. There is no Push Notification support.

### 2. Technical Limitations
*   **Storage (Base64):** User avatars are currently stored as Base64 strings directly in MongoDB. While convenient for a POC, this increases database size and memory overhead.
*   **Socket.IO Scaling:** The current implementation uses an in-memory Socket.IO adapter. It will not work across multiple server instances without a Redis adapter.
*   **Message Caching:** The frontend caches messages in memory (Zustand). There is no persistent client-side storage (like IndexedDB), so a refresh triggers a full re-fetch.

---

## 📈 Future Scope & Scaling

To transition this from a POC to a production-grade system, the following roadmap is proposed:

*   **Database Scaling:** Transition to a sharded MongoDB cluster and implement **Redis Caching** for frequently accessed user profiles and active conversation metadata.
*   **WebSocket Performance:**
    *   Implement **Redis Adapter** for Socket.IO to enable horizontal scaling across multiple pods.
    *   Transition from `Socket.IO` to raw **WebSockets** or **gRPC** for even lower overhead in high-throughput scenarios.
*   **Binary Storage:** Move avatar and image storage from MongoDB to a dedicated object store like **AWS S3** or **Google Cloud Storage** with a CDN (CloudFront) layer.
*   **Message Durability:** Implement a "Message Delivery Status" (Sent -> Delivered -> Read) with a queue-based system (RabbitMQ/Kafka) to ensure messages are never lost during server restarts.
*   **Security:** Implement Rate Limiting, Helmet.js for header security, and multi-factor authentication (MFA).
